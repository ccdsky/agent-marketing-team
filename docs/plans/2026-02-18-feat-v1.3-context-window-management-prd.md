# v1.3: Context Window Management

## Overview

Every subagent spawned in Campaign Mode reads 7–9 files before doing any work. Those reads happen independently for every agent, even when multiple agents are running on the same campaign simultaneously. The context window fills with setup overhead before a single word of output is produced.

v1.3 addresses this with a **Campaign Context Snapshot** — a single, Campaign Lead-maintained file that distills all operational context into one read. Agents consult the snapshot instead of independently synthesizing from source files. Source files remain authoritative but become reference documents, not required reading.

This is not a cosmetic change. It is an architectural one that determines whether the system scales to complex campaigns without context exhaustion.

---

## Problem Statement

### The Current Per-Agent Loading Pattern

When Campaign Lead spawns a Research Specialist or Creative Specialist subagent in Campaign Mode, the agent follows the Pre-Task Protocol defined in TEAM.md:

```
Before claiming any task, read:
1. context/voice-dna.md
2. context/icp.md
3. context/business-profile.md
4. output/campaigns/[slug]/campaign-brief.md
5. knowledge/research/[topic]-[date].md (if available)
6. knowledge/learnings/campaigns/ (Grep for patterns)
7. .claude/skills/[skill]/SKILL.md (the relevant skill)
```

Plus the agent reads its own definition file and TEAM.md before any of this.

**Conservative token estimates per agent invocation:**

| File | Estimated Tokens |
|------|-----------------|
| TEAM.md | ~800 |
| Agent definition file | ~1,200 |
| context/voice-dna.md | 500–2,000 (user-defined) |
| context/icp.md | 500–2,000 (user-defined) |
| context/business-profile.md | 400–1,500 (user-defined) |
| campaign-brief.md (mid-campaign) | 800–2,000 |
| Research file | 1,500–4,000 |
| Skill file | 800–1,500 |
| **Total** | **~6,500–15,000 tokens** |

That is the cost of *setup* before a single word of output. In a Campaign Mode sprint where Campaign Lead spawns three specialists in parallel, this overhead is paid three times — once per agent — reading many of the same files.

### Why This Matters More Over Time

The problem compounds as campaigns progress:

- **Sprint 1:** Research files don't exist yet. Overhead is manageable.
- **Sprint 2:** Research files are large. Campaign brief has grown with Sprint 1 decisions. Overhead increases.
- **Sprint 3:** Multiple research files. Full campaign brief. Multiple prior drafts to reference. Overhead at its peak — precisely when context efficiency matters most.

The current architecture gets more expensive at the moment when agents need the most remaining context for creative work.

### What Compounding Was Designed to Solve (and Didn't Fully Solve)

The `knowledge/learnings/` compounding approach correctly addresses *cross-campaign* memory: rather than re-reading a 3,000-word research document from a prior campaign, agents Grep for condensed learnings. That is working as designed.

What it does not address is *intra-campaign per-agent setup overhead* — the cost of every agent reading the same current campaign's source files from scratch, every invocation. Compounding reduces cross-campaign reads; it does not reduce within-campaign reads.

---

## Goals

1. **Reduce per-agent setup overhead by 70–85%** — from 7–9 file reads to 1 primary read
2. **Preserve correctness** — agents must have accurate, current operational context
3. **Keep source files authoritative** — voice-dna.md, icp.md, campaign-brief.md remain the ground truth; the snapshot summarizes, not replaces
4. **Campaign Lead owns synthesis** — no new agent role; Campaign Lead already reads all source files and is the right owner of distilled context
5. **Simple Mode unaffected** — this is a Campaign Mode optimization; standalone asset requests work exactly as today

## Non-Goals

- Eliminating all file reads (agents will still fetch source files when task-specific detail is needed)
- Solving cross-campaign memory (already addressed by `knowledge/learnings/`)
- Building a general-purpose vector memory or RAG system (that is v2.0)
- Changing the sprint model or task coordination architecture

---

## Proposed Solution: The Campaign Context Snapshot

Campaign Lead creates and maintains a single file per campaign:

```
output/campaigns/[slug]/context-snapshot.md
```

This file contains the **operationally relevant distillation** of all source context, synthesized by Campaign Lead at campaign kickoff and updated at each sprint transition.

**The key design principle:** The snapshot contains only facts agents need to make decisions and produce output. Full detail lives in source files and is fetched only when a specific task genuinely requires it.

### What the Snapshot Is Not

- Not a replacement for skill files (agents still read the relevant SKILL.md)
- Not a replacement for task metadata (deliverable paths still flow via TaskUpdate)
- Not a substitute when an agent needs deep detail (e.g., a Quality Gate agent checking voice needs to read actual voice-dna.md writing samples, not just the operational summary)

---

## Update Protocol

### When Campaign Lead Creates/Updates the Snapshot

| Trigger | What Changes |
|---------|-------------|
| Campaign kickoff | Create initial snapshot from context files |
| Sprint 1 checkpoint approved | Add Active Positioning Angle + Strategic Decisions |
| Sprint 2 kickoff | Add/update Asset Map |
| New strategic decision made | Update Strategic Decisions section |
| Research completes | Update Research Summary section |
| Sprint 3 kickoff | Update Asset Map with all draft paths |

**Campaign Lead never delegates snapshot maintenance.** It is their core coordination artifact.

### Snapshot vs. Campaign Brief

These are complementary, not redundant:

| | Campaign Brief | Context Snapshot |
|-|---------------|-----------------|
| **Purpose** | Strategic record — what we decided and why | Operational context — what agents need to execute |
| **Audience** | User + Campaign Lead | Specialist subagents |
| **Length** | Long — captures full history | Short — distilled for token efficiency |
| **Updated by** | Campaign Lead | Campaign Lead |
| **Format** | Narrative + decisions log | Structured for fast agent reads |

The campaign brief remains the authoritative strategic record. The snapshot is the efficient operational extract.

## Agent Protocol Changes

### TEAM.md Pre-Task Protocol (updated)

**Current:**
```
Before claiming any task, read:
- context/voice-dna.md
- context/icp.md
- context/business-profile.md
- Campaign brief: output/campaigns/[slug]/campaign-brief.md
- Relevant research: knowledge/research/[topic]-[date].md
- Past learnings: knowledge/learnings/campaigns/
```

**Proposed:**
```
Before claiming any task, read:
- output/campaigns/[slug]/context-snapshot.md (primary — all operational context)

Read source files only when the task specifically requires more detail:
- context/voice-dna.md — if task requires voice analysis or writing sample examples
- knowledge/research/[topic]-[date].md — if task requires specific data points
- output/campaigns/[slug]/campaign-brief.md — if task requires strategic history

Past learnings: knowledge/learnings/campaigns/ (unchanged — cross-campaign patterns)
```

### Per-Agent Changes

Each agent definition file's Pre-Task Protocol section collapses from a 6-item list to:

```
Read the campaign context snapshot at:
output/campaigns/[slug]/context-snapshot.md

Fetch source files only when your specific task requires detail beyond the snapshot.
```

This is a one-line change per agent file, but it changes the economics of every Campaign Mode invocation.

## Phase Design

### v1.3 — Campaign Context Snapshot (This PRD)

**Scope:** Add snapshot creation to Campaign Lead kickoff workflow. Update Pre-Task Protocol in TEAM.md and all agent files. Define the snapshot format and update triggers.

**Implementation effort:** Moderate. Mostly prompt changes + new Campaign Lead workflow step.

**Risk:** Low. Additive change. Source files remain authoritative. If the snapshot is missing or incomplete, agents can fall back to current behavior.

**Testable outcome:** Spawn a Campaign Mode specialist subagent and observe: does it read 7–9 files or 1 file before claiming a task?

---

### v2.0 — Selective Memory (Future Phase)

The snapshot is a human-curated distillation maintained by Campaign Lead. This works but has a ceiling: Campaign Lead must decide what to include, and decisions can be wrong.

A more sophisticated approach — inspired by how OpenClaw and similar systems manage working memory — would make context **queryable rather than pre-loaded**:

**Concept:** Instead of reading a single snapshot file, agents issue a semantic query against a structured memory index:

```
ContextQuery("ICP pain points for this campaign")
→ Returns: 200-token distillation of the 3 most relevant pain point clusters

ContextQuery("voice patterns — sentence structure")
→ Returns: 150-token operational voice note with examples
```

The memory system answers with only the facts relevant to the query — not the full context of everything that might be relevant. This mirrors how QMD-style systems in production agent architectures handle working memory: fetch what's needed for the next action, not everything that might ever be needed.

**Why this is v2.0, not v1.3:**
- Requires building a memory indexing and retrieval layer
- Requires defining the query interface for agents
- Snapshot approach validates the concept and measures token savings first
- v1.3 data (how much context do agents actually need?) informs v2.0 design

**Prior art to study:**
- Compound Engineering's `knowledge/learnings/` pattern — already implements a rudimentary "write once, query by Grep" memory
- OpenClaw memory management — how the system decides what to retain vs. evict between agent turns
- Production RAG patterns for agent systems — chunking, indexing, and relevance-gated retrieval

## Open Questions

1. **Snapshot granularity:** How much voice nuance can be compressed without losing what makes the owner's voice distinctive? ICP pain points compress well. Voice patterns may not — some owners' voices live in specific sentence rhythms that resist summarization.

2. **Fallback protocol:** If context-snapshot.md doesn't exist (e.g., early in Sprint 1 before Campaign Lead has created it), do agents fail-safe to the current multi-file read? The spec should define this explicitly.

3. **Snapshot versioning:** Should the snapshot overwrite in-place (simpler, but loses history) or version by sprint (snapshot-s1.md, snapshot-s2.md)? Versioning aids debugging; overwriting is simpler for agents to find.

4. **Quality Gate exception:** Quality Gate agents evaluate voice fidelity and need actual writing samples from voice-dna.md, not just the operational summary. The protocol should explicitly tell Quality Gate to always fetch voice-dna.md in full, not rely on the snapshot for this.

5. **Research-in-progress state:** In Sprint 1, research tasks and drafting tasks sometimes run in parallel. If Research Specialist hasn't completed yet, the snapshot's Research Summary section is empty. Does Campaign Lead publish a partial snapshot and update it, or wait until research completes?
